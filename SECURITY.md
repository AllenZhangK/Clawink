# Security Policy

Clawink is currently in a **public preview** stage.

The public repository is intentionally limited, and the core product code is still being opened in stages. Even so, if you believe you have found a security issue related to the public materials, release process, or any future public code slice, please report it responsibly.

## Please Do Not Report Security Issues Publicly

Do **not** open a public Issue, Discussion, or pull request that includes:

- Vulnerability details
- Exploit steps
- Sensitive configuration findings
- Security-sensitive screenshots or logs

Public disclosure before review creates unnecessary risk.

## How To Report A Security Issue

At the current stage, please report security issues **privately** to the repository owner through GitHub.

Recommended approach:

1. Go to the repository owner profile
2. Use an available private contact channel
3. Include enough detail to reproduce and assess the issue safely

If a dedicated security contact address is added later, this file will be updated.

## What To Include

A useful security report usually includes:

- A clear description of the issue
- The affected surface or file
- Preconditions required to trigger the issue
- Reproduction steps
- Expected impact
- Any suggested mitigation, if known

## Response Expectations

During the preview stage:

- Reports are reviewed manually
- Response time is best effort
- Fixes may land in the internal repository first and appear later through public sync

Not every report will lead to an immediate public patch, especially while the public code surface is still limited.

## Scope Notes

At this stage, the highest-priority reports are:

- Problems in public release content
- Exposure risks in the one-way sync process
- Sensitive information accidentally included in public assets or docs
- Security-relevant flaws in any future public runtime slice

Thank you for handling security findings responsibly.
